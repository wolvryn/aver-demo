# ADR-001: Demo design — the Aver agent output integrity showcase

- **Status:** Accepted
- **Date:** 2026-06-06
- **Deciders:** Rob Wisniewski
- **Repo:** `aver-demo` (public)
- **Related:** `wolvryn-docs` ADR-001 (repositioning), `wolvryn-docs` ADR-002 (Aver service offering)
- **Lineage:** Revives and supersedes the parked Telemetry / agent-edge monitoring project (its ADR-001), now built with Butler as the parts bin rather than from scratch.

---

## Context

The org-level ADRs established Aver as a **non-invasive instrument that reads an agent pipeline's observable output** (never its internals) to assess output integrity, and named the demo as the **critical path**: the hero proof that makes the claim concrete for buyers and anchors the rebuilt site. The demo cannot be sold on a claim alone, and it must be *honest* — it has to reproduce the real engagement condition, in which Aver attaches to foreign code that was never written with Aver in mind.

This ADR settles the demo's architecture, its build order relative to Aver, the fault-class scope and run model, and the ingestion contract the demo emits — which is also Aver's front-door contract.

## Decision

### Build order

1. **Build the demo pipeline first; build Aver after it.** The pipeline is built as a standalone application that is **entirely unaware of Aver**. This forces an honest external attach boundary: Aver must observe a genuine piece of foreign code from the outside, exactly as in a real engagement. (This supersedes an earlier, incorrect "the demo imports Aver" framing and its implied lockstep build.)

### Architecture — three pieces, one hard boundary

2. The demo comprises three distinct components:
   - **The pipeline** (two lanes) — Aver-unaware; reuses Butler's collect → AI-summarize pattern.
   - **Aver** — attaches to and reads only the pipeline's **observable output surface**; it never touches pipeline internals. This black-box boundary is the invariant and the embodiment of the positioning.
   - **The dual-pane UI** — lives in `aver-demo`; the only component that orchestrates both pipeline and Aver to render the split view (pipeline output on one side, Aver's verdict on the other).

3. **Two lanes.** Lane A = "good": collect from an API → AI summarize → clean output. Lane B = the same shape with injected faults.

### Faults and run model

4. **v1 scope: all four fault classes** — ungrounded fabrication, cross-source contradiction, stale-data roll-forward, entity attribute-bleed. The full taxonomy is the differentiation, and the "all faults at once" showstopper requires all four. They share one ingestion contract, so the marginal cost of each detector after the first is low.
   - **Build sequence within v1 is vertical-slice, not big-bang:** fabrication first (proves the attach seam end-to-end), then the remaining three one at a time.
   - **Injected faults are calibrated to be legible** — realistic but unambiguous. The demo proves Aver catches representative faults convincingly; it is not a benchmark against worst-case detection difficulty. This protects v1 from the two genuinely hard detectors (contradiction, attribute-bleed).

5. **Fault injection is controllable, with modes: single / all / random.** This lets a demo be *driven as a narrative* — walk one fault cleanly, then trigger the all-at-once showstopper — while preserving live credibility via the random mode.

6. **Live, not scripted.** Generation runs live per run. The honest technique: **inject the fault into the source/input data, then let the live agent genuinely produce the faulty output from it** — never script a fake output. The fault emerges from the agent, as it would at a client.
   - Live carries real costs, amplified because `aver-demo` is public and is the site hero: per-visit model calls (cost, latency, non-determinism) and an abuse surface. Guardrails to design in: capped / rate-limited runs, and controlled injection so the buyer-facing run always lands. Live generation also reactivates the deferred demo-hosting question (serverless execution limits vs. a live pipeline → reuse Butler's chunked/heartbeat orchestration pattern).

### The ingestion contract (shared with Aver's front door)

7. **The demo emits the canonical Aver ingestion surface.** The pipeline's output surface is a clean reference implementation of the ingestion record, doubling as the canonical example of "the minimum Aver needs from you."

8. **Record shape:**
   - **Core:** `output` + `sources[]` — sources is a **set** from the start (cross-source contradiction needs at least two).
   - **Enrichment slots, each demand-derived from a specific fault class** (no speculative fields): **timestamps** (source as-of + output produced-at → stale roll-forward), **entity references** (→ attribute-bleed), and a **run/trace ID** (correlate and group records). The schema grows only when a new fault class arrives needing a new field.
   - **Adapters** normalize whatever the customer emits — OTel GenAI spans, raw JSON logs, a DB audit table, a captured API stream — into this normalized record. This is Butler's adapter-factory pattern: vendor-named adapters feeding one standardized contract.

9. **Three-state per-fault reporting.** Every fault check returns one of: **clean**, **fault found**, or **cannot-assess — missing signal X**. A missing enrichment field must never cause a silent skip; "cannot-assess" is a first-class finding. The set of cannot-assess results is a signal-maturity / coverage map handed to the customer, and it routes into a precursor "get-you-emitting-the-signal" engagement rather than a dead end. (Detailed detection behavior lives in the `aver` repo's own ADRs; it is summarized here because it shapes the contract and the demo's story.)

10. **Answer-key discipline.** The demo injects the faults, so it knows ground truth. That ground truth is used **only to score Aver** — caught N of N, false positives, misses — producing a far stronger demo than "Aver flagged some things." The answer key is **never** fed to Aver; Aver detects blind, mirroring real conditions where no answer key exists.

## Consequences

**Positive**
- The Aver-unaware pipeline produces a truthful attach boundary; the demo proves Aver against foreign code, not Aver-fitted code.
- The demo's output surface doubles as the canonical ingestion example and a buyer qualification artifact ("can you give me, per output, the output and its sources?").
- Scoring against the held-back answer key yields a decisive demo ("7 injected, 7 caught, 0 false alarms").
- Heavy reuse of Butler patterns (collect/summarize pipeline, adapter factory) — the parts-bin play.

**Negative / costs to reckon with**
- Live generation means ongoing model cost, non-determinism, and a public abuse surface; requires rate-limiting and run controls.
- The demo's compute home is still unresolved (see Open items).
- All-four scope is more than a minimal proof; mitigated by vertical-slice build order and legible (non-adversarial) injected faults.

## Alternatives considered

- **Build Aver and the demo in lockstep / demo imports Aver.** Rejected: produces bespoke coupling and a dishonest demo that proves Aver only against code fitted to it. Demo-first with an Aver-unaware pipeline is the truthful condition.
- **Scripted or replayed demo.** Rejected: forfeits the "it is really running" credibility that is core to the proof. Live is chosen despite its cost.
- **v1 with a single fault class.** Rejected: undersells the taxonomy (the differentiation) and cannot deliver the all-faults showstopper. Vertical-slice build is retained *inside* the all-four scope to get the same incremental safety.
- **Invasive instrumentation (wrapping pipeline steps).** Rejected per the org positioning: non-invasive output-reading is the honest, sellable mechanism — "we don't make you rewrite your agents."

## Open items (non-blocking; tracked for follow-on)

- **Demo compute / hosting** — serverless execution limits vs. a live pipeline; its own decision.
- **Aver internal detection architecture**, per fault class — `aver` repo ADR(s).
- **Source-retention reality and the precursor engagement** (helping a customer emit the integrity signal when they lack it) — a service-level decision, likely an addendum to `wolvryn-docs` ADR-002.
