# ADR-002: Public/private exposure boundary — how the demo invokes Aver

- **Status:** Accepted
- **Date:** 2026-06-06
- **Deciders:** Rob Wisniewski
- **Repo:** `aver-demo` (public)
- **Related:** `aver-demo` ADR-001 (demo design); `wolvryn-docs` ADR-001/002 (repositioning, service offering)

---

## Context

`aver-demo` is a **public** repository and the site's hero proof. `aver` is **private** and contains Wolvryn's proprietary detection logic — the actual value of the offering. ADR-001 said the demo "consumes Aver" but did not specify *how* without exposing proprietary code. That invocation boundary is exactly where exposure would happen, so it is settled here.

The requirement is absolute: the public repo, and anything it ships to a browser, must contain **nothing proprietary** and must not reveal a Wolvryn process that lives in a private repo.

## Decision

1. **Aver is invoked as a separate deployed service over HTTP**, server-side, from `aver-demo`. The demo POSTs the ingestion record to the Aver service and renders the returned verdicts.

2. **The public repo contains zero Aver detection code.** Its Aver-related contents are limited to: a thin **server-side Aver client module** (owns the HTTP call) and the **shared ingestion-contract types**.

3. **Contract is public; detection is secret.** The ingestion record shape is the customer-facing "minimum Aver needs" and is safe in the open. *How Aver judges a record* never enters the public repo or the browser bundle.

4. **No lifted private source.** Butler and other private repos inform *patterns only*. No Butler/private source, internal infrastructure references, or Wolvryn-internal process details are copied into this repo. The demo pipeline is built clean and generic.

5. **Server-side only.** The Aver endpoint and any credentials live server-side; the browser never calls Aver and never sees the endpoint. The answer key stays demo-side and is never sent to Aver.

6. **Scope boundary — this is the *demo's* topology, not the engagement model.** Invoking a hosted Aver service is correct for the demo (Wolvryn's own public, non-sensitive data). It is **not** a commitment to how real engagements are deployed. The real-engagement deployment and data-residency model (hosted vs. in-customer-network) is a separate, unsettled decision — see Open items. Aver must therefore be built as a **deployment-agnostic unit** (a container runnable anywhere); "hosted service" is one topology, not the only one.

## Consequences

**Positive**
- The public repo *provably* contains nothing proprietary — protection does not depend on perpetual discipline against accidental imports.
- Clean, inspectable boundary; the detection logic has exactly one home (the private `aver` service).
- Matches the black-box model: the demo hands Aver a record and gets a verdict.

**Negative / costs**
- Requires a deployed Aver service and adds a network hop.
- The shared contract types need a home both repos can use (define them public-side here, or publish a small shared contract package — a minor follow-on choice).

## Alternatives considered

- **Bundle/import Aver source into `aver-demo`.** Rejected outright — puts proprietary detection logic in a public repo.
- **Aver as a server-only private dependency.** Rejected — the public repo would not contain the source, but the deployed build would, and a single accidental client-side import leaks it to the browser. Protection would depend on discipline holding forever.
- **Aver as a called service.** Chosen — the only option where the public repo contains no proprietary code by construction.

## Open items (non-blocking; tracked for follow-on)

- **Engagement deployment & data-residency model** (hosted Aver vs. Aver deployed inside the customer network, taking only findings out) — a real-engagement decision with direct bearing on FinTech sellability; likely a future `wolvryn-docs` ADR. Build Aver deployment-agnostic so this stays open.
- **Shared contract package vs. public-side type definitions** — minor; decide when Aver's repo needs the types.
